TERMS & CONCEPTS

master || main || trunk || develop
is what opens by default on github
is verified from rest of team
becomes production
"a permanent branch which always reflects a production-ready state"

branch
copy of the master
used to work on individual features/bugfixes/chores desired to be pushed up to the master
"represents an independent line of development | serves as an abstraction for the edit/stage/commit process in Git"

branch naming convention
  • chore - changing simple things like a name
  • feature - new compononet
  • bugfix- broken code that needs to be fixed

    • hotfix / quick-fix
    fixes up a production bug fast and is applied

    COMMANDS

    to create a new branch
    git checkout -b feature/blog

    to go to branch
    git checkout feature/blog

    to delete branch
    git branch -d feature/blog

    to see all branches
    git branch

    *
    denotes current branch

    "feature/chat" - just a naming convention - just a string - not a containing folder
    be careful making new branches because it's based on where you are right now - the current branch
    always go back to master and make the new thing - should always be based on the master - should always create a branch off of the master
    PULL REQUEST
    git pull
    commonly referred to as "PR"
    is to grab all the updated version from the master on github making local copy up to date with one published on github

    MERGE vs. REBASE

    • NEW COMPONENTS
    • rebase maintains history
    • merge messes up history

    CONFLICTS

    git stash save

    • temporary stash
    • do not restart computer or it will delete it

    git stash pop

    • bring it back out to work on
    • similar to rebase and merging
    • is used for working with stuff locally
